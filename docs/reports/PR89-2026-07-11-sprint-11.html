📁 +3 arquivos · ➕ 124 linhas · ➖ 1492 linhas

⚡ ~2150 tokens

#88 #90 refatoração CI/CD

Antes

EntryForm monolítico com 699 linhas e 18 campos, usando condicionais isTransfer para mostrar/esconder seções de transferência. Entradas.tsx dependia de transferInitialData + handleOpenTransfer para abrir transferência, forçando troca de aba e pré-seleção de tipo de origem.

Conceito "Clube" poluía formulários com checkbox + meses + auto-recorrência ao selecionar tipo de origem com hasRecurrence: true.

CI executava 2 vezes no mesmo commit (push + pull_request) sem controle de concorrência.

Depois

3 formulários dedicados independentes, cada um com apenas os campos relevantes. TransferForm: origem → destino, data, pontos, bônus, carrinho — sem tipo de origem, sem recorrência. EntryFormPontos/Milhas: tipo de origem, recorrência manual — sem campos de transferência.

"Clube" removido completamente: recorrência sempre manual via checkbox. Backward compatível — entradas existentes com recurrenceInterval preservadas na edição.

CI com concurrency para cancelar execução duplicada no mesmo branch.

Benefícios

  • ✅ -1368 linhas líquidas (net) — EntryForm + Clube removidos
  • ✅ Cada formulário sabe seu propósito — sem condicionais isTransfer
  • ✅ TransferForm independente — sem trocar aba ou pré-selecionar tipo
  • ✅ EntryFormPontos/Milhas focados — sem campos de transferência poluindo
  • ✅ Clube removido — recorrência sempre manual via checkbox
  • ✅ Backward compat — entradas existentes com recurrenceInterval preservadas
  • ✅ CI sem duplicação — concurrency cancela execução anterior no mesmo branch
  • ✅ EntryFormData em types/ — compartilhado entre forms + recurrence.ts
  • ✅ 54/54 E2E + 45/45 unit — todos verdes
