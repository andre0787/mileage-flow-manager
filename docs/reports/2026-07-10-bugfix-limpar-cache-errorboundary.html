🐛 Bugfix: Limpar Cache disparava ErrorBoundary

2026-07-10 | bug fix

Sintoma

Ao clicar em "Limpar Cache" na aba Preferências de /configuracoes, o app exibia:

"Algo deu errado
Ocorreu um erro inesperado ao carregar esta seção. Tente recarregar a página ou clique no botão abaixo."

🔍 Root Cause (2 bugs encadeados)

Bug #1: queryClient.clear() antes do reload

1. Usuário clica "Limpar Cache"
2. queryClient.clear() — invalida TODAS as queries
3. React re-renderiza com cache vazio → queries refetch
4. entries é undefined — TypeError no OrigemTypeSection
5. ErrorBoundary captura o TypeError
6. window.location.reload() — recarrega a página (tarde demais)

Bug #2: entries não desestruturado em Configuracoes.tsx

O Configuracoes.tsx fazia:

const { owners, programs, origemTypes, accounts, clearCache, clearAccountData, isLoading } = useData();

Mas usava entries na prop de OrigemTypeSection:

<OrigemTypeSection origemTypes={origemTypes} entries={entries} ... />

Como entries não estava na desestruturação, era undefined. Quando o componente chamava entries.some(...), lançava TypeError → ErrorBoundary.

Esse erro existia SEMPRE, mas só era visível quando o React re-renderizava após queryClient.clear(). Nas navegações normais, o cache da SPA evitava o re-render completo.

🔧 Fixes

Fix #1: DataContext.tsx — remove queryClient.clear()

const clearCache = () => {
const keys = Object.keys(localStorage)
.filter(k => k.startsWith("mc-") || k === "mc-migrated");
keys.forEach(k => localStorage.removeItem(k));
queryClient.clear();
window.location.reload();
};

Por quê: reload() já descarta cache in-memory. queryClient.clear() antes só introduzia um estado inconsistente.

Fix #2: Configuracoes.tsx — adiciona entries na desestruturação

const { owners, programs, origemTypes, accounts, entries, clearCache, clearAccountData, isLoading } = useData();
