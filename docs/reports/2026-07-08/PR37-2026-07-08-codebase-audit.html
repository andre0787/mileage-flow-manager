🔴 Críticos

C1 — strictNullChecks desligado

tsconfig.json tem strictNullChecks: false. Isso significa que null e undefined passam despercebidos pelo compilador e só aparecem em runtime. É a principal fonte de crashes do tipo Cannot read properties of undefined.

// tsconfig.json
"strictNullChecks": false  ← DESLIGADO
"noImplicitAny": false     ← DESLIGADO
"noUnusedParameters": false
"noUnusedLocals": false

💡 Ação: ativar strict: true gradualmente, corrigindo os erros um por um.

C2 — Zero cobertura de testes unitários

O projeto tem 8 testes E2E (Playwright) mas zero testes unitários. src/lib/metrics.ts tem 209 linhas de funções PURAS perfeitas para unit test — mas nenhuma delas é testada individualmente. Toda a lógica financeira (cálculo de lucro, margem, ROI, custo médio) depende exclusivamente dos testes E2E para ser validada.

📊 Funções sem teste:

  • calcCostPerMile, calcCostPerThousand, calcAverageCostPerMile
  • calcProfit, calcProfitMargin, calcROI
  • calcMilesGenerated, calcProportionalCost
  • calcWeightedAverageCost, calcRevenueChange
  • filterActiveSales, filterSalesByMonth
  • computeDashboardMetrics

💡 Ação: adicionar vitest + testes para todas as funções em src/lib/.

C3 — DRY violado: Vendas.tsx recalcula profit/margin/ROI inline

src/pages/Vendas.tsx (1117 linhas) NÃO importa calcProfit, calcProfitMargin ou calcROI de src/lib/metrics.ts. Em vez disso, recalcula tudo inline em 3 lugares:

1. Simulador (linhas 106-113):

const profit = saleValue - totalCost;
const margin = saleValue > 0 ? (profit / saleValue) * 100 : 0;
const roi = totalCost > 0 ? (profit / totalCost) * 100 : 0;

2. JSX do formulário (linhas 578-582):

R$ {(... saleValue - miles*costPerMile - additionalCost).toFixed(2)}
{((... / saleValue) * 100).toFixed(1)}%

💡 Ação: substituir por calcProfit(), calcProfitMargin(), calcROI() de metrics.ts.

🟠 Alta Prioridade

A1 — Entradas.tsx (1276 linhas) num único function component

É o maior arquivo do projeto. Tudo está dentro de export default function Entradas() — sem subcomponentes, sem extração de hooks customizados, sem separação de responsabilidades. Mistura: renderização de 2 abas (Milhas/Pontos), formulário de criação, formulário de edição, lógica de transferência, lógica de carrinho, busca, resumo de métricas, resumo por conta.

💡 Ação: extrair EntryForm, EntryTable, EntrySummary, TransferForm, CartForm em componentes separados.

A2 — Vendas.tsx (1117 linhas) no mesmo problema

Segundo maior arquivo. Mesmo padrão: tudo em um function component. Mistura: tabela de vendas com 2 visões (mobile/desktop), formulário de criação, simulador em modal, métricas do header, busca + filtro.

💡 Ação: extrair SaleForm, SaleTable, SaleSimulator, SaleMetrics em componentes separados.

A3 — useDatabase.ts com 40 funções de 7 entidades

822 linhas, 40 funções exportadas. Mistura queries + mutations de owners, programs, origem_types, accounts, entries, sales, clients. Cada mutation repete o mesmo padrão (declara queryClient, faz insert/update/delete, invalida queries).

💡 Ação: quebrar em useOwners.ts, useAccounts.ts, useEntries.ts, useSales.ts, useClients.ts etc.

💡 Opcional: criar helper createMutation para eliminar o boilerplate repetitivo.

A4 — Lint errors não corrigidos (3 erros) 
src/types/index.ts:116   error  Empty block statement  (catch {})
tests/fluxo-completo.spec.ts:219  error  'mCmb' → use 'const'
tests/fluxo-completo.spec.ts:285  error  'vCmb' → use 'const'

O catch {} vazio engole erros de parse silenciosamente. Os let deveriam ser const.

💡 Ação: catch { return {}; } e let → const.

A5 — Formatação numérica espalhada (dezenas de toLocaleString)

.toLocaleString('pt-BR') e .toFixed() aparecem dezenas de vezes em Entradas.tsx e Vendas.tsx — cada um com parâmetros diferentes. Não há uma utility centralizada formatCurrency() ou formatNumber().

💡 Ação: centralizar em src/lib/utils.ts:

export function formatCurrency(value: number): string
export function formatNumber(value: number): string
export function formatPercent(value: number): string

🟡 Média Prioridade

M1 — Configuracoes.tsx (751 linhas) — terceiro maior

Agrupa CRUD de 4 entidades (Donos, Programas, Origens, Contas) no mesmo arquivo. Cada entidade tem seu próprio estado, validação e formulário — tudo inline.

💡 Ação: extrair OwnerSection, ProgramSection, OrigemTypeSection, AccountSection.

M2 — supabase-types.ts (299 linhas) — tipagem gigante gerada

Arquivo de tipos gerados automaticamente do Supabase. É esperado que seja grande, mas poderia ser movido para src/types/supabase.ts e simplificado com z.infer do Zod em vez de depender de tipos gerados.

💡 Ação: manter como está se for gerado automaticamente, ou migrar para Zod.

M3 — Testes E2E duplicam setup de dados

Cada spec.ts faz seu próprio page.evaluate() para criar dados via REST API, com headers, URLs e lógica duplicados. O tests/helpers.ts existe mas parece subutilizado — os testes criam os próprios helpers inline.

💡 Ação: centralizar a criação de dados de teste em helpers.ts e reutilizar entre specs.

M4 — Sem Prettier ou formatador padronizado

Não há .prettierrc nem configuração de formatação automática. O estilo do código depende exclusivamente dos autores. Isso leva a inconsistências (espaços vs tabs, aspas simples vs duplas, ponto-e-vírgula vs sem).

💡 Ação: adicionar Prettier + .prettierrc + script format.

🟢 Baixa Prioridade

B1 — src/lib/accounts.ts pouco utilizado (1 import)

O módulo tem funções de conta que só são importadas em 1 lugar. Pode indicar que a lógica de domínio migrou para as páginas ou hooks.

💡 Verificar se o módulo ainda é necessário ou se o código foi duplicado nas páginas.

B2 — src/lib/accounts.ts e origemTypes.ts com 71 e 33 linhas

Módulos de domínio muito pequenos. Podem ser fundidos em src/lib/domain.ts ou expandidos com a lógica que hoje está inline nas páginas.

B3 — ErrorBoundary em toda rota (9x em App.tsx)

Cada rota em App.tsx wrappa o componente com <ErrorBoundary>. Poderia ser aplicado uma vez no nível do layout.

// Em vez de repetir 9x:
<Route path="/x" element={<ErrorBoundary><X /></ErrorBoundary>} />
// Usar um layout wrapper:
<Route element={<ErrorBoundaryLayout />}> ... </Route>
B4 — AnimatedNumber só usado no Dashboard

Componente de animação numérica está em src/components/AnimatedNumber.tsx mas só é usado no Dashboard. MetricCard tem sua própria lógica interna também.

💡 Poderia ser incorporado ao MetricCard ou mantido como está.

B5 — testes/responsivo.spec.ts (283 linhas) — maior test file

Teste responsivo é o maior arquivo de teste. Verifica 11 páginas × 4 viewports. É valioso mas poderia ser otimizado com fixtures do Playwright para evitar repetição.
